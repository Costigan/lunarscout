## Required before expanding the examples

1. Reconcile the plan with the implemented API

   Do one focused inventory pass and update inaccurate status claims. In particular:
    - Weighted scoring already exists in 18_map_algebra_screening.py.
    - Focal cleanup and basic distance fields already exist in 19_map_algebra_focal.py.
    - Temporal reduction/composition already exists in 20_map_algebra_temporal.py.
    - Terrain, resampling, and bounded writes already exist in
      21_map_algebra_terrain_resample.py.

    - Unit-bearing power and several “remaining eager gaps” appear to be implemented.
    - Clearly mark all large-raster focal, regional, distance, zonal, and temporal execution as
      deferred.

   This prevents unnecessary implementation work based on stale checkboxes.

2. Fix correctness inconsistencies in operations the examples will use

   Limit this audit to the example-facing eager surface:
    - dtype inference and promotion;
    - integer overflow behavior, especially unsigned and uint64;
    - invalid fill and nodata representability;
    - numeric-error handling for division, logarithms, square roots, and casts;
    - accumulator behavior in focal/global/zonal reductions;
    - unit propagation and validation.

   Full internal centralization is desirable, but only observable inconsistencies need to block
   examples.

3. Audit registry claims for the supported example surface

   Make describe_operation() and list_operations() accurately report:
    - eager versus file-backed availability;
    - parameters and defaults;
    - dtype, unit, and validity behavior;
    - connectivity, edge, resampling, distance, and approximation choices.

   Registry completeness for operations not used or advertised by the examples can wait.

4. Make explain() and plan() adequate for the review-before-run example

   They need to expose the scientifically meaningful choices actually used in that workflow:
    - source paths and identities;
    - thresholds and inclusivity;
    - units;
    - validity behavior;
    - explicit alignment/resampling;
    - output grid, dtype, and encoding;
    - supported execution mode and window plan.

   They do not need a general cost model, backend analysis, local fusion report, or temporary-
   disk estimator.

5. Add focused public-API tests for those contracts

   Test through import lunarscout as ls in a fresh process, covering:
    - valid zero versus invalid pixels through GeoTIFF round trips;
    - grid and unit mismatch;
    - exact integer limits, including uint64;
    - non-finite and all-invalid inputs;
    - eager/expression parity for operations used in examples;
    - QGIS-oriented Boolean encoding and dataset masks;
    - explain() and plan() being read-only;
    - unsupported file-backed operations failing during planning, before output creation.

   This is the last implementation gate before treating the example-facing API as stable.
